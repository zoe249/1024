# 微信好友榜接入文档

## 目标

本文档只讨论 **微信小游戏好友排行榜** 的实现方案，不包含全服排行榜。

结论先说清楚：
- **只做微信好友榜，不需要自建服务器**
- **也不要求必须接入云开发**
- 依赖的是微信小游戏提供的 **开放数据域** 和 **用户云托管数据** 能力

这套方案适合当前项目，因为你的需求是展示“当前用户和微信好友之间的分数排名”，不涉及全服排序、运营活动榜、赛季结算和防作弊后台。

## 适用范围

适用于以下场景：
- 查看当前用户的微信好友分数排行
- 在游戏内弹出好友榜面板
- 展示当前用户自己的排名、昵称、头像、分数
- 后续扩展群排行

不适用于以下场景：
- 全服排行榜
- 跨平台排行榜
- 活动榜、赛季榜
- 强校验、防刷分、防作弊后台统计

如果未来要做这些能力，就需要补云开发或服务器。

## 核心概念

### 1. 主域

主域就是你当前这个正式游戏工程。

它负责：
- 游戏逻辑
- UI 交互
- 分数计算
- 打开排行榜面板
- 向开放数据域发送消息
- 把开放数据域绘制出来的内容显示在主域界面里

对应你当前项目，主域就是现在这套 Cocos Creator 工程。

### 2. 开放数据域

开放数据域是一个 **独立的小游戏子工程**，运行环境和主域隔离。

它负责：
- 读取微信好友关系链数据
- 读取微信云托管分数数据
- 在 `sharedCanvas` 上绘制排行榜
- 接收主域消息，切换显示内容

微信好友数据相关接口，通常应该放在开放数据域里使用，而不是直接在主域里做完整渲染逻辑。

### 3. 用户云托管数据

好友榜的分数不是你本地排，而是先把用户自己的分数上报到微信托管数据，再由开放数据域读取好友数据进行排序显示。

常用做法：
- 主域在合适时机调用 `wx.setUserCloudStorage`
- 存储一个排行榜 key，例如 `score`
- value 必须是字符串
- 开放数据域调用 `wx.getFriendCloudStorage`
- 拿到好友分数后本地排序并绘制

## 整体方案

整体结构建议拆成两部分。

### 主域部分

建议主域负责以下模块：
- `WechatRankService.ts`
- `RankPanelController.ts`

职责建议：
- `WechatRankService.ts`
  - 封装微信平台判断
  - 封装分数上报
  - 封装打开/关闭排行榜消息发送
  - 封装主域与开放数据域通信
- `RankPanelController.ts`
  - 控制排行榜弹窗开关
  - 管理显示区域
  - 控制遮罩、关闭按钮、面板动画
  - 在打开时通知子域开始绘制

### 开放数据域部分

建议单独建一个开放数据域工程，或者在现有微信构建流程中挂接开放域项目。

开放数据域建议至少包含：
- `index.js / game.js` 入口
- 排行榜数据管理模块
- 排行榜绘制模块
- 主域消息监听模块

职责建议：
- 监听主域消息
- 拉取好友分数数据
- 进行排序和格式化
- 绘制列表、头像、昵称、分数、排名
- 处理翻页或滚动（如果以后需要）

## 数据流

推荐的数据流如下：

1. 游戏正常运行，`PlayController` 负责算出当前分数
2. 当分数刷新到新高时，主域调用 `wx.setUserCloudStorage`
3. 用户点击“排行榜”按钮
4. 主域打开排行榜弹窗，并通过 `wx.getOpenDataContext().postMessage(...)` 通知开放数据域
5. 开放数据域收到消息后，调用 `wx.getFriendCloudStorage`
6. 开放数据域对好友数据排序
7. 开放数据域把排行榜绘制到 `sharedCanvas`
8. 主域将 `sharedCanvas` 内容显示在排行榜面板中

## 分数上报策略

### 建议只上报最高分

好友榜通常建议只上报一个字段：
- key: `score`
- value: 当前历史最高分字符串

原因：
- 排行榜只关心可比性，不需要上传所有局内状态
- 字段越少，后续越稳定
- 未来如果要扩展，也可以再增加 `level`、`maxTile` 之类的 key

### 推荐上报时机

建议按下面优先级处理：
- 游戏结束时上报一次
- 刷新历史最高分时立即上报一次
- 从后台回前台时，如果本局数据有变化，可补一次上报

不建议每一帧都上报。

原因：
- 没必要
- 会增加无意义调用
- 也不利于排查问题

### 推荐存储格式

推荐：
- key: `score`
- value: `"12345"`

注意：
- value 必须是字符串
- 排序时要转成数字再比大小

## 对当前项目的落地建议

基于你当前项目结构，建议这样接。

### 1. 分数来源

你当前已经有分数系统，排行榜分数建议直接取：
- 当前总分 `Score`
- 历史最高分 `BestScore`

好友榜上报建议用：
- `BestScore`

原因：
- 排行榜通常显示历史最佳成绩
- 用户体验更稳定
- 不会因为一局中途退出导致榜单数值来回跳

### 2. 主域新增模块建议

建议在 `assets/script/` 下新增：
- `WechatRankService.ts`
- `RankPanelController.ts`

建议职责如下。

`WechatRankService.ts`
- 判断是否在微信小游戏环境
- 维护排行榜 key 名称
- 比较本地最高分和待上报分数
- 上报好友榜分数
- 向开放数据域发送 `showFriendRank`、`hideFriendRank`、`refreshFriendRank` 消息

`RankPanelController.ts`
- 控制排行榜弹窗显示和关闭
- 打开时调用 `WechatRankService` 请求绘制好友榜
- 关闭时通知子域停止刷新
- 管理面板动画、遮罩和关闭按钮

### 3. 开放数据域消息协议建议

主域与子域不要直接散写消息名，建议统一约定协议。

建议消息结构：

```ts
{
  type: 'showFriendRank',
  payload: {
    scoreKey: 'score'
  }
}
```

可以预留这些消息：
- `showFriendRank`：显示好友榜
- `refreshFriendRank`：刷新好友榜
- `hideFriendRank`：隐藏好友榜
- `updateSelfScore`：主域上报成功后，通知子域主动刷新自己数据

这样后面扩展群排行榜时，只需要再补：
- `showGroupRank`

## 开放数据域渲染建议

排行榜 UI 建议全部由开放数据域自行绘制，不要依赖主域里的普通节点树去拼。

原因：
- 开放数据域最终输出的是 `sharedCanvas`
- 主域看到的是一张实时画布贴图
- 主域里不能像普通节点那样直接访问子域内部 UI 节点

建议开放数据域绘制内容包括：
- 排名序号
- 用户头像
- 用户昵称
- 分数
- 自己的高亮样式

建议先做静态列表，不要一开始就做复杂滚动。

第一版做法：
- 固定显示前 20 名
- 超出先不处理

这样成本最低，也最稳。

## Cocos 接入建议

### 1. 主域显示方式

在 Cocos 中，一般做法是：
- 主域创建一个排行榜面板节点
- 面板内放一个用于显示开放数据域内容的区域
- 打开排行榜时，让这个区域显示 `sharedCanvas`

具体实现方式要结合 Cocos Creator 3.8 的微信小游戏构建能力来接。

### 2. 构建方式

你需要准备两套构建结果：
- 主游戏构建到微信小游戏
- 开放数据域构建到微信小游戏开放域

Cocos 官方文档里有专门的“开放数据域”章节，建议按该流程配置。

### 3. 排行榜面板建议

建议你在主场景里单独放一个排行榜弹层，例如：
- `RankOverlay`
  - `Mask`
  - `Panel`
  - `CloseButton`
  - `SubContextView`

其中：
- `Mask` 负责半透明遮罩
- `Panel` 负责视觉容器
- `SubContextView` 或对应显示区域负责呈现 `sharedCanvas`

## 推荐开发顺序

建议按下面顺序推进，避免一次做太多导致不好排查。

### 第一阶段：先跑通上报

先只做主域：
- 平台判断
- 调用 `wx.setUserCloudStorage`
- 上报一个固定测试分数

目标：
- 确认微信后台能接收到该用户分数

### 第二阶段：再跑通子域读取

再做开放数据域：
- 监听主域消息
- 调用 `wx.getFriendCloudStorage`
- 把结果打印出来或简单绘制出来

目标：
- 确认能拿到好友数据

### 第三阶段：最后做面板 UI

最后再接主域排行榜面板：
- 打开/关闭动画
- 遮罩
- `sharedCanvas` 显示区域
- 排行榜样式美化

这样拆开以后，出了问题更容易定位。

## 常见坑

### 1. 浏览器预览看不到好友榜

这是正常的。

原因：
- 微信好友关系链能力只在微信小游戏环境下有效
- 浏览器预览通常无法真实返回这类数据

所以排行榜相关功能要重点在：
- 微信开发者工具
- 真机预览

### 2. `value` 不是字符串会失败

`wx.setUserCloudStorage` 的 `value` 需要是字符串。

错误示例：
- 直接传数字 `123`

正确做法：
- 转成 `"123"`

### 3. 好友榜为空

常见原因：
- 当前账号没有玩过该小游戏的好友
- 好友没有上报过对应 key 的数据
- key 名写错
- 子域读的 key 和主域写的 key 不一致

### 4. 头像、昵称、列表样式在主域里不好控制

这是开放数据域方案的特点。

因为排行榜内容本质上是在子域画布里绘制，主域只能显示结果，不能像普通节点那样逐个控制内部元素。

所以建议：
- 排行榜 UI 尽量在子域内部自己绘制完整
- 主域只负责“容器”和“开关”

### 5. 子域资源和主域资源不能想当然共用

开放数据域是相对独立的运行环境。

所以：
- 子域要用到的图资源、字体资源、布局代码，要按子域自己的工程来准备
- 不要假设主域里的资源路径，子域一定能直接拿到

### 6. 排行榜模糊

这是 `sharedCanvas` 常见问题。

一般要检查：
- 画布尺寸是否按设备像素比处理
- 显示区域和实际绘制尺寸是否一致
- 是否发生了二次拉伸

第一版建议先保证功能正确，再处理清晰度。

## 是否需要云开发或服务器

针对当前这个“微信好友榜”需求，结论很明确：
- **不需要自建服务器**
- **也可以先不接云开发**

因为：
- 好友榜依赖微信关系链数据
- 分数上报和好友读取都可以走微信小游戏开放能力
- 排序和绘制在开放数据域内完成即可

只有当你后续要做：
- 全服榜
- 活动榜
- 赛季榜
- 强防作弊

才建议补后端。

## 推荐的第一版范围

为了尽快落地，第一版建议只做这些：
- 主域上报 `score`
- 开放数据域读取好友榜
- 固定展示前 20 名
- 主域有一个排行榜弹窗
- 可以打开和关闭
- 可以看到自己和好友的分数顺序

先把链路打通，再考虑：
- 自己高亮
- 滚动列表
- 群排行榜
- 动画效果
- 样式美化

## 后续可扩展方向

这套方案以后可以平滑扩展：
- 增加 `maxTile` 字段，用于比较最大合成数字
- 增加 `bestScore` 和 `seasonScore` 两套 key
- 增加群排行榜
- 在主域里增加“我的排名”摘要卡片
- 增加排行榜奖励展示，但奖励结算如果要防作弊，仍建议放后端

## 实施建议

如果下一步开始正式落地，我建议按下面顺序做：

1. 主域补一个 `WechatRankService.ts`，先把上报能力接好
2. 新建开放数据域工程，先把好友数据拉取和简单绘制跑通
3. 主场景增加 `RankOverlay`，把排行榜面板接进来
4. 最后再做样式和交互细节
