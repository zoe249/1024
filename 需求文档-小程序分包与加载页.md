# 需求文档：小程序分包与进入游戏加载页

## 一、背景

当前项目是一个基于 Cocos Creator 3.8.8 的竖屏休闲游戏，后续计划发布到小游戏/小程序平台。为了优化首次启动速度、控制主包体积，并改善玩家从启动到进入游戏的体验，需要优先完成以下两项能力建设：

1. 小程序分包方案
2. 进入游戏前的加载页面

本文档用于明确这两个需求的目标、范围、交互和验收标准，作为后续开发依据。

## 二、需求目标

### 2.1 总体目标

- 降低首包压力，提升小游戏首开速度
- 避免玩家在黑屏或无反馈状态下等待
- 在进入主游戏场景前，向玩家明确展示加载进度和当前状态
- 为后续增加更多场景、资源、广告、音频和活动内容预留扩展空间

### 2.2 本阶段目标

- 完成适合当前项目体量的小程序分包规划
- 增加一个独立的进入游戏加载页
- 支持从启动页进入加载页，再进入主游戏场景
- 在加载页中完成主游戏所需资源的预加载或必要初始化

## 三、需求范围

### 3.1 本次包含

- 小程序/小游戏平台下的分包规划
- 启动后进入加载页的流程设计
- 加载页 UI 与状态文案
- 资源预加载逻辑
- 加载失败时的基础提示与重试机制

### 3.2 本次不包含

- 登录系统
- 排行榜、分享、广告等平台能力接入
- 热更新系统
- 新玩法或新关卡内容
- 完整的新手引导系统

## 四、需求一：小程序分包

### 4.1 需求目标

将项目内容按“首次启动必须内容”和“进入后按需加载内容”拆分，减少主包体积，确保玩家尽快进入可交互流程。

### 4.2 分包原则

- 主包只保留启动必需资源和代码
- 首次进入游戏所需的最小资源集应尽量集中
- 非首屏资源延后加载
- 后续扩展内容应具备继续拆分为独立分包的能力

### 4.3 建议拆分方案

#### 主包建议包含

- 启动入口脚本
- 加载页场景
- 加载页所需图片、字体、基础音效
- 进入主游戏前必须执行的初始化逻辑
- 主游戏场景跳转逻辑

#### 首个功能分包建议包含

- 主游戏场景 `game.scene`
- 棋子 Prefab
- 主玩法脚本
- 主游戏运行必须资源

#### 后续可扩展分包

- 结算页、活动页、皮肤页
- 广告相关资源
- 音频资源包
- 特效资源包
- 未来新增玩法场景

### 4.4 功能要求

1. 项目需要输出一份明确的分包资源清单。
2. 代码中需要区分主包启动逻辑与游戏内容加载逻辑。
3. 启动后默认先进入加载页，而不是直接进入主游戏场景。
4. 加载页需负责拉起主游戏分包或主游戏资源加载。
5. 分包失败时，需要提示用户并支持重试。

### 4.5 实现约束

- 方案应兼容 Cocos Creator 3.8.8 当前导出能力
- 不应破坏现有 `game.scene` 及玩法逻辑结构
- 不应为了分包而大面积改动核心脚本结构
- 尽量保持资源目录清晰，避免未来扩展时重复迁移

## 五、需求二：进入游戏加载页

### 5.1 需求目标

在玩家启动小游戏后，先展示一个加载页，用于承接启动过程中的等待时间，并在完成必要资源加载后进入主游戏。

### 5.2 页面定位

加载页是启动后的第一个可见页面，其职责包括：

- 展示品牌或游戏名称
- 提示当前加载状态
- 展示加载进度
- 执行进入主游戏前的预加载
- 加载完成后自动跳转主游戏

### 5.3 页面内容要求

加载页建议至少包含以下元素：

- 游戏名称或 Logo
- 背景图或基础视觉底图
- 进度条
- 百分比数字
- 当前状态文案

状态文案示例：

- 正在初始化...
- 正在加载游戏资源...
- 即将进入游戏...
- 加载失败，请点击重试

### 5.4 交互要求

1. 用户启动小游戏后，立即看到加载页。
2. 页面出现时，不应长时间黑屏。
3. 加载过程中，进度条和文案应有变化反馈。
4. 加载完成后，自动进入主游戏场景。
5. 加载失败后，页面展示失败提示和重试按钮。

### 5.5 功能要求

1. 加载页应支持资源预加载进度展示。
2. 若底层无法提供精确进度，至少需要提供阶段性进度反馈。
3. 进入主游戏前，必须保证主游戏核心资源已可用。
4. 加载完成后的跳转过程应稳定，不出现重复跳转。
5. 重试逻辑应避免重复创建多个加载流程实例。

### 5.6 视觉要求

- 风格与当前游戏轻量休闲风格保持一致
- 适配竖屏移动端
- 视觉结构简洁，优先保证清晰反馈
- 页面资源尽量轻量，避免加载页自身过重

## 六、推荐流程

建议整体流程如下：

1. 小程序启动
2. 进入加载页场景
3. 执行基础初始化
4. 加载主游戏分包或主游戏核心资源
5. 更新进度条与状态文案
6. 加载完成后进入 `game.scene`
7. 若失败则提示并允许用户重试

## 七、技术实现建议

### 7.1 场景建议

- 新增单独的加载页场景，例如 `loading.scene`
- 该场景放在主包中
- `game.scene` 作为后续进入的主游戏场景

### 7.2 代码建议

- 新增加载页控制脚本，例如 `LoadingController.ts`
- 将资源预加载、分包加载、跳转逻辑集中在加载页控制脚本中
- 主游戏逻辑仍保留在现有 `PlayController.ts`

### 7.3 资源建议

- 加载页仅使用少量基础资源
- 不要把大量主游戏资源仍然打入主包
- 主游戏资源应尽量跟随场景或分包统一管理

## 八、验收标准

### 8.1 分包验收

- 存在明确的分包方案与资源归属说明
- 启动流程已不再直接进入主游戏场景
- 主游戏内容可以通过加载页流程正常进入
- 分包或资源加载失败时，页面可以提示并重试

### 8.2 加载页验收

- 启动后可见加载页
- 加载页包含进度反馈
- 加载完成后稳定进入 `game.scene`
- 不出现明显黑屏、卡死或重复跳转
- 加载失败时可手动重试并恢复正常流程

## 九、开发优先级

建议按以下顺序开发：

1. 新增加载页场景与基础 UI
2. 调整启动流程，先进入加载页
3. 接入主游戏资源预加载与进度显示
4. 梳理主包与分包资源归属
5. 增加加载失败与重试处理
6. 最后再做视觉优化与加载细节体验优化

## 十、待确认项

以下问题建议在正式开发前确认：

1. 目标平台是微信小游戏、抖音小游戏，还是多个平台共用
2. 当前包体积是否已经接近平台限制，是否需要强制优先做分包
3. 加载页是否需要展示品牌 Logo 或版权信息
4. 是否需要在加载页中接入音频初始化
5. 后续是否计划加入活动页、商城页、结算页等适合拆分分包的内容
